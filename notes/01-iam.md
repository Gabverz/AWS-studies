# 01 – IAM: Identity and Access Management

These notes summarize the key concepts of AWS IAM, covering users, groups, roles, policies, and access best practices.

---

## 1. What is IAM?

AWS IAM (Identity and Access Management) is the service used to control **who can access AWS** and **what they are allowed to do**.

It is a **global service** — IAM configurations apply across all regions in an account.

Main resources managed by IAM:

- **Users** – individual identities (human or application)
- **Groups** – collections of users that share permissions
- **Roles** – identities assumed by services, applications, or external users
- **Policies** – documents that define allowed or denied actions

---

## 2. Root Account

The root account is the identity created when the AWS account is first opened.

Characteristics:

- uses the email address used to register the account
- has **full and unrestricted access** to all AWS resources and settings
- cannot be limited by IAM policies

Best practices:

- enable **MFA immediately** after account creation
- do **not** use the root account for daily operations
- do **not** create access keys for the root account
- use it only for:
  - account-level billing settings
  - changing support plans
  - account closure

---

## 3. IAM Users

An IAM user is an identity created inside the AWS account for a specific person or application.

Characteristics:

- has a name and credentials (password and/or access keys)
- can access the AWS Console (with password) and/or the API/CLI (with access keys)
- permissions are defined by attached policies or group membership

Best practices:

- create one IAM user per person
- never share credentials between users
- grant only the permissions needed for the task (least privilege)
- rotate access keys regularly
- disable or delete users that are no longer needed

---

## 4. IAM Groups

A group is a collection of IAM users that share the same permissions.

How it works:

- policies are attached to the **group**, not directly to each user
- users in the group **inherit** all group permissions
- a user can belong to **multiple groups**
- groups **cannot** contain other groups

Example groups:

| Group Name      | Purpose                              |
|-----------------|--------------------------------------|
| Administrators  | Full access for account management   |
| Developers      | Access to specific services for dev  |
| ReadOnly        | View-only access to resources        |
| DataEngineers   | Access to S3, Glue, Redshift, etc.   |

Best practice: manage permissions through groups, not by attaching policies directly to individual users.

---

## 5. IAM Roles

A role is an IAM identity that can be **assumed** by a user, a service, or an external identity.

Unlike users, roles:

- do **not** have long-term credentials (password or access keys)
- issue **temporary credentials** when assumed
- are commonly used for:
  - EC2 instances accessing S3 or other services
  - Lambda functions calling AWS APIs
  - cross-account access
  - identity federation (SSO, Active Directory, etc.)

Common example:

- an EC2 instance needs to read from an S3 bucket
- instead of hardcoding credentials, attach an **IAM role** with S3 read permissions to the instance
- the instance automatically gets temporary credentials to access S3

---

## 6. IAM Policies

A policy is a **JSON document** that defines:

- **Effect** – Allow or Deny
- **Action** – which AWS API actions are covered
- **Resource** – which specific resources the rule applies to
- **Condition** – optional conditions (IP range, MFA requirement, etc.)

Example of a simple policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

Types of policies:

- **AWS Managed Policies** – created and maintained by AWS, ready to use
- **Customer Managed Policies** – created by you, more specific control
- **Inline Policies** – attached directly to a single user, group, or role (not reusable)

How policies are evaluated:

- by default, **everything is denied**
- an explicit **Allow** is required for access
- an explicit **Deny** always overrides an Allow

---

## 7. MFA (Multi-Factor Authentication)

MFA adds a second factor to the login process, protecting accounts even when passwords are compromised.

How it works:

- after entering the password, a **one-time code** is required
- the code is generated by an **authenticator app** (e.g., Google Authenticator, Authy)
- the code expires after a short time

Where to enable:

- **root account** – always, without exception
- **IAM users with console access** – especially those with elevated permissions

MFA can also be required by policy as a **condition** for specific actions.

---

## 8. Access Keys

Access keys are used to authenticate programmatic access (CLI, SDK, API).

Structure:

- **Access Key ID** – public identifier (like a username)
- **Secret Access Key** – private key (like a password, shown only once)

Best practices:

- never share access keys
- never commit them to code repositories
- rotate keys regularly
- prefer **IAM roles** over access keys whenever possible
- delete keys that are no longer in use

---

## 9. Least Privilege Principle

The principle of least privilege means granting **only the permissions required** to perform a specific task — nothing more.

In practice:

- start with no permissions
- add only what is needed for the task
- review permissions regularly and remove what is no longer needed

This reduces the impact if a credential is compromised or misused.

---

## 10. IAM Best Practices Summary

| Practice                               | Why it matters                              |
|----------------------------------------|---------------------------------------------|
| Enable MFA on root account             | Protects the most powerful identity         |
| Do not use root for daily work         | Reduces exposure of the highest-privilege account |
| Create individual IAM users            | Enables traceability and accountability     |
| Use groups to manage permissions       | Easier to maintain at scale                 |
| Apply least privilege                  | Limits blast radius of a compromised credential |
| Use roles for services and automation  | Avoids long-term credentials in code        |
| Rotate access keys regularly           | Reduces the risk of credential exposure     |
| Review permissions periodically        | Removes unnecessary access over time        |
