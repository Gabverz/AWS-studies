# 02 – Compute: Amazon EC2

These notes summarize the key concepts of Amazon EC2 and the main steps practiced when creating a first EC2 instance.

---

## 1. What is Amazon EC2?

Amazon EC2 (Elastic Compute Cloud) is AWS's virtual server service. It allows you to create and manage compute instances in the cloud with full control over:

- operating system
- CPU and memory
- storage
- networking
- access configuration

EC2 is one of the core building blocks of AWS and is commonly used when a workload needs:

- a full operating system
- custom software installation
- direct control over the environment
- integration with other AWS services such as S3, EBS, RDS, and IAM

---

## 2. What is an EC2 Instance?

An EC2 instance is a **virtual machine running in AWS**.

It behaves similarly to a physical server, but it is created on demand and can be resized, stopped, or terminated as needed.

Each instance is associated with:

- a **VPC**
- a **subnet**
- one or more **security groups**
- at least one **EBS volume** for storage
- optionally, an **IAM role**
- optionally, a **public IP address**

---

## 3. Common Use Cases

EC2 is commonly used for:

- web servers
- application servers
- APIs
- development and test environments
- batch processing
- custom workloads that require OS-level control
- workloads that need integration with other AWS services

---

## 4. Instance Types and Families

AWS offers multiple instance families, each optimized for a different type of workload.

| Family | Optimized For | Example Use Cases |
|---|---|---|
| General Purpose | Balanced CPU and memory | Web apps, small services, dev environments |
| Compute Optimized | High CPU performance | Batch jobs, CPU-heavy applications |
| Memory Optimized | Large in-memory workloads | Databases, caches, analytics |
| Storage Optimized | High disk throughput and IOPS | Data-intensive workloads |
| Accelerated Computing | GPU / specialized hardware | ML, graphics, video processing |

Common beginner-friendly instance types:

- `t3.micro`
- `t3.small`

These are often used for learning, testing, and low-cost environments.

## 5. Main Components of an EC2 Launch

When creating an EC2 instance, several parameters must be defined.

### 5.1 AMI (Amazon Machine Image)

The AMI defines the operating system and base software for the instance.

Common examples:

- Amazon Linux
- Ubuntu
- Windows Server

### 5.2 Instance Type

The instance type defines the available:

- CPU
- memory
- network performance
- storage performance

For study and small labs, it is usually best to start small.

### 5.3 Key Pair

A key pair is used to authenticate access to the instance.

- Linux instances usually use **SSH**
- Windows instances usually use **RDP**

Important:

- the private key file is downloaded only once
- AWS does not store the private key for you
- it must be kept in a safe place

### 5.4 Network Settings

Network configuration includes:

- VPC selection
- subnet selection
- public IP assignment
- security group association

This determines whether the instance will be reachable from the internet.

### 5.5 Storage

The root disk of the instance is typically an **EBS volume**.

You can configure:

- volume size
- volume type
- additional attached volumes

### 5.6 Advanced Details

Optional advanced settings may include:

- **IAM role**
- **user data**
- monitoring options
- shutdown behavior

## 6. Security Groups

A **Security Group** acts as a virtual firewall for the EC2 instance.

Key characteristics:

- it controls **inbound** and **outbound** traffic
- it is **stateful**
- it allows traffic only through explicitly defined rules
- it does not support deny rules

### Common Inbound Rules

| Protocol | Port | Typical Source | Purpose |
|---|---|---|---|
| SSH | 22 | Your IP only | Linux remote access |
| RDP | 3389 | Your IP only | Windows remote access |
| HTTP | 80 | 0.0.0.0/0 | Public web traffic |
| HTTPS | 443 | 0.0.0.0/0 | Secure web traffic |

Best practices:

- restrict SSH and RDP to **your IP only**
- avoid exposing administrative ports to the entire internet
- create separate security groups for different tiers when needed

## 7. How to Create an EC2 Instance

The basic launch flow in the AWS Console is:

1. Open **EC2**
2. Go to **Instances**
3. Click **Launch instances**
4. Select the required parameters
5. Review the configuration
6. Launch the instance

Typical steps during launch:

- choose an AMI
- choose an instance type
- configure the key pair
- define networking options
- configure the security group
- configure storage
- launch the instance

## 8. How to Access an EC2 Instance

### 8.1 Linux Instances via SSH

Basic SSH connection pattern:

```bash
chmod 400 your-key.pem
ssh -i "your-key.pem" ec2-user@<public-ip-or-dns>
```

Default usernames may vary depending on the AMI:

- Amazon Linux → `ec2-user`
- Ubuntu → `ubuntu`
- Debian → `admin`

### 8.2 Windows Instances via RDP

For Windows instances:

- download the RDP file from the AWS Console
- decrypt the administrator password using the key pair
- connect using an RDP client

### 8.3 Session Manager

AWS Systems Manager Session Manager is a more secure alternative for access.

Benefits:

- no need to open SSH or RDP ports
- no need to manage public keys for interactive access
- access is controlled through IAM permissions
- better auditing and security for production environments

## 9. EC2 and EBS

EC2 instances usually rely on **EBS volumes** for storage.

Important points:

- EBS is **block storage**
- the root volume is usually an EBS volume
- volumes remain available even if the instance is stopped
- volumes continue to generate cost while they exist

Typical use cases for EBS with EC2:

- root filesystem
- application data
- logs
- temporary processing storage
- snapshots for backup or cloning

---

## 10. EC2 Lifecycle

Understanding lifecycle states helps avoid unnecessary costs.

| State | Description | Billing |
|---|---|---|
| Pending | Instance is starting | No compute charge yet |
| Running | Instance is active | Compute + EBS |
| Stopping | Instance is shutting down | EBS only |
| Stopped | Instance is powered off | EBS only |
| Terminated | Instance is permanently deleted | No charge |

Best practice:

- stop instances when you do not need them
- terminate test instances when they are no longer required
- remove unused EBS volumes when possible

---

## 11. Cost Awareness

EC2 cost depends mainly on:

- instance type
- instance size
- running time
- attached EBS storage
- outbound data transfer

Ways to reduce cost:

- choose the smallest suitable instance type
- stop or terminate instances not in use
- delete unused EBS volumes and snapshots
- avoid overprovisioning for lab environments
